# real-world-sim 测试说明（含状态日志解读）

本文对应脚本：
- `test/real-world-sim.js`
- `test/real-world-sim-worker.js`

运行命令：

```bash
node test/real-world-sim.js
```

该脚本是“长时压测/场景模拟”，不是“单次断言型单元测试”。它会持续循环执行，直到手动停止。

## 1. 这个 test 测了什么

`real-world-sim` 模拟了一个完整机房课堂场景，覆盖：

1. 批量账号与教室准备  
创建教师、教室、学生（默认 10 教师 x 5 教室 x 100 学生 = 5000 学生端）。

2. 大规模 WS 接入与绑定  
通过 10 个 worker 并发创建 WS 客户端并执行 `bind`。

3. 课堂期间持续通信  
每 30 秒心跳一次；每 2 分钟随机做重连和违规上报；教师端按时间表下发广播、开关守卫、远程 kill、白名单更新等操作。

4. 课堂结束清理  
断开 WS、销毁 worker、删除房间和教师，进入下一轮循环。

## 2. 状态日志行字段解释

日志形态：

```text
[INFO] [状态] 连接 A(-B) | 心跳 X/Y (R%) | 重连 C | 违规 V | 运行 Tmin
```

字段含义：
[2026-04-25T10:36:49.240Z] [INFO] [状态] 连接 4502(-498) | 心跳 108002/302 (0.3%) | 重连 550 | 违规 1143 | 运行 12min
| 字段 | 含义 | 统计口径 |
|---|---|---|
| `连接 A(-B)` | A = 连接+绑定成功总数；B = 连接/绑定失败总数 | 来自 connect 阶段 worker 汇总，后续不会按真实在线数实时回落 |
| `心跳 X/Y (R%)` | X = 累计发送心跳数；Y = 累计收到 `heartbeat-ack` 数；R = `Y/X` | 累计值，不是当前 30 秒窗口值 |
| `重连 C` | 累计重连成功次数 | 每 2 分钟触发一次随机重连后累加 |
| `违规 V` | 累计违规上报次数 | 这里记的是上报消息次数，不是违规明细条数 |
| `运行 Tmin` | 当前循环已运行分钟数 | `floor((now - loopStart)/60000)` |

为什么会“打印很多这个”：

- 课堂阶段会固定每 30 秒打印一次状态（直到课堂阶段结束），所以这是预期行为，不是异常刷屏。

## 3. 你这段日志的结果解读

你给出的片段核心特征：

1. `连接 4502(-498)`  
计划 5000 连接，最终成功 4502、失败 498，连接成功率约 90.04%。

2. `心跳 ... (0.0%~0.1%)`  
例如 `64490/88 (0.1%)`，心跳 ack 极低，说明“发送很多、确认极少”，课堂链路稳定性很差。

3. `重连` 在增长（100 -> 305）  
说明每 2 分钟的重连流程确实在执行并有部分成功。

4. `违规` 在增长（187 -> 630）  
说明违规上报流程在执行。

5. `[广播(开始)] 推送至 50 个房间, 0 台子机`  
表示广播 API 请求成功执行，但当时房间内可发送对象统计为 0（没有可发送的在线 WS 客户端）。

## 4. 这次运行的结论（按当前日志）

这轮不能视为“课堂通信健康”：

1. 接入阶段只有约 90% 成功（5000 目标下偏低）。  
2. 心跳 ack 持续接近 0%，明显异常。  
3. 广播 sent=0，说明控制消息无法有效触达在线端。  

## 5. 代码口径与已知限制（影响你读数）

1. `连接 A(-B)` 不是实时在线数，而是连接阶段累计结果。  
2. `心跳 X/Y` 是累计值，不是当前 30 秒窗口值。  
3. 当前脚本没有“自动判定通过/失败”阈值，主要靠日志人工判读。  

## 6. 本次现象的可疑根因（基于现有实现）

以下是代码层面的高风险点，会导致你看到“连接看起来很多，但 ack 极低、广播为 0”：

1. `wsConnect` 的超时定时器没有在连接成功后清理，可能在超时点关闭已连接 socket。  
位置：`test/real-world-sim-worker.js`（`wsConnect`）。

2. 客户端连接对象没有统一在 `ws close` 事件里回写 `connected=false`，统计可能与真实在线状态脱节。  
位置：`test/real-world-sim-worker.js`（客户端状态维护逻辑）。

3. 心跳 ack 等待 TTL 较短（3 秒），高并发下会放大 ack 丢失统计。  
位置：`test/real-world-sim-worker.js`（`heartbeatAll` + `wsWait`）。

## 7. 建议的“健康运行”判定线（供后续对比）

建议把以下指标作为基线（可按机器性能调整）：

1. 连接成功率 >= 98%  
2. 心跳 ack 比率 >= 95%  
3. 课堂广播 sent / 在线数 >= 80%  
4. 重连成功率（成功/尝试）>= 80%

如果任一指标长期不达标，应判定该轮“压测失败/课堂链路不健康”。
